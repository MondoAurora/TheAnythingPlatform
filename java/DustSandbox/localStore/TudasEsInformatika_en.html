When we talk about IT development, we often use terms from "architecture" and "architectural design".

And the interesting thing about this is that the first buildings looked something like this.

So we tried to build structures from approximately homogeneous, interlocking elements.

It had practically no internal structure or plan.

It is simply the laws of physics themselves that hold the pyramid shape together.

When we start building purposefully, it becomes clear that we are working very seriously to precisely shape the individual elements and stick them to each other.

We can build much higher with much less material.

However, these materials still look like we're stacking stones on top of each other and gravity is basically holding this building together, and if there's an earthquake, for example, its stability is quite questionable.

With this methodology, we can build enormous buildings from a human perspective, they are very tall compared to a person, and we can cover large distances and dimensions with them.

But if the question is what technology enables us, then we need to change our methodology.

An example of this is the Eiffel Tower, in which case the building elements are not only held together by practical connections and physical forces, i.e. gravity, but they are also able to withstand tensile forces at individual points.

And effectively, the Eiffel Tower is a physical manifestation of a mathematical formula, essentially, how can we safely distribute the mass of this structure so that it stays standing like this.

We are still talking about software architecture, right? In my opinion, this is how our software design methodology works, namely, that we tightly connect the modules through APIs, function calls, etc.

There is one problem with this, or there are several problems.

One is that its mass will be terrible.

The other is that it reacts terribly poorly to changes.

Well, we can't practically change this Eiffel Tower.

If we want to change this, we need to look at two things.

One is how much mass is absolutely necessary to reach a given height.

This is the Snelson needle tower.

Between the two, from an architectural perspective, Eiffel speaks to integrity.

So that each element meets the other element in the same way and at the same angle.

This is secured by screws and rivets.

The Snelson model conveys a way of thinking called Tensegrity, which is about how individual elements can withstand either pressure or pull.

We never connect compression-bearing elements and tension-bearing elements directly, but rather use a flexible connection between them, meaning, compression-bearing elements are only connected to tension-bearing elements.

This is practically translated into computer science into late binding, dynamic loading and alike.
